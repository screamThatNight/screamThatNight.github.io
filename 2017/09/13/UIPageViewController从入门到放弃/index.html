UIPageViewController从入门到放弃

发表于 2017-09-13

UIPageViewController

分页控制器可以让用户在页码之间导航，这些页码都是被各自的视图控制器管理。
导航可以被你的app或者用户使用的手势程序化的控制。
当在页面间导航时，页面控制器使用你指定的过渡动画来过渡页面。

开始

当定义一个页面控制器界面时，你可以一次提供一个内容视图控制器（或者一次两个，取决于spine position 以及 double-sized状态）或者在需要的情况下使用data source。当一次提供一个内容视图控制器时，你应当使用setViewControllers:direction:animated:completion:方法来设置当前的内容视图控制器数组。如果想要支持基于手势的导航，你必须使用数据源对象提供视图控制器。
页面浏览控制器的数据源负责根据需要提供内容视图控制器并且必须符合UIPageViewControllerDataSource协议。UIPageViewControllerDelegate协议提供一些与外观相关的信息并接收关于手势发起的转换的通知。

property and methods desc

setable property desc
dataSource 响应于基于手势的导航来调用数据源的方法。 如果此属性的值为nil，则禁用基于手势的导航。
delegate 为了响应基于手势的导航以及方向的变更会调用代理内的方法
doubleSided 如果页面的后面没有内容（值为NO），则页面正面的内容将在翻页时部分显示到后面。如果脊椎位于中间，则值必须为YES。 将脊椎置于中间时将其设置为NO会引发异常。

代理解析

Data Source

先从数据源代理开始，数据源实现可以以任何适合你app的方式自由的处理responsibility,在许多情况下，它应该查看传递给它的视图控制器，确定需要被显示的内容，并且根据需要创建他们需要的视图控制器。你会发现它对于在视图控制器中包含一些额外信息比如页码等，简化要展示的内容任务很有帮助.
如果Supporting a Page Indicator ( - presentationCountForPageViewController: && - presentationIndexForPageViewController: )这两个数据源代理方法都被实现，并且该Page VC 的transition style是 UIPageViewControllerTransitionStyleScroll，那么一个page indicator就是可见的。
这些方法会在setViewControllers:direction:animated:completion:方法调用后，被调用。
在手势驱动的导航后，这些方法不会被调用。这个index会自动更新并且视图控制器的数量预计将保持不变。

Delegate

代理能接收到设备方向的变化以及用户导航到新页面的时机。
对于page_curl过渡模式，代理可以提供一个不一样的spine location来对应界面方向的更改。

我的结论


真特么鸡肋。。。
首先说说使用过程中遇到的问题，初始化PageViewController时，无论是采用Curl还是Scroll，都必须调用setViewControllers...这个方法，如果不给予PageViewController的初始内容子内容视图，其后续就再也不会新的界面了，为什么呢？
因为DataSource代理方法的触发依赖于用户的操作！！！

由于存在这样的局限性，显而易见，我们没办法在恰当的时机手动增加子视图控制器，如果你坚持任务重复调用 setViewControllers这个方法，那我也没办法，因为这样的做法是很愚蠢的。
再者，第二个巨大的局限性是，我们没办法跳页，我想看到第100页，就只能一页一页的翻过去。。。
这简直是判了死刑了喂。
适配的场景太少了，也难怪用它的人很少。
一言以蔽之，放弃，转战自定制的PageViewController。
